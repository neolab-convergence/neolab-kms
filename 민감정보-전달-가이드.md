# 민감 정보 안전 전달 가이드

NeoLAB KMS 인계 시 두 가지 민감 파일을 안전하게 전달해야 합니다. 이 문서는 **무엇을, 어떤 채널로, 어떻게** 전달할지 정리합니다.

## 🔐 전달해야 할 민감 파일

### 1. `.env` (운영 서버 환경변수)
- 위치: 운영 서버 `/opt/neolab-kms/.env`
- 내용: Google OAuth secret, session secret, admin 이메일 목록 등
- 노출 시 위험: 누구나 KMS 관리자로 인증 가능, 세션 위조 가능

### 2. `deploy-key-infra-kms` (운영 서버 SSH 비밀키)
- 위치: 본인 PC `~/.ssh/deploy-key-infra-kms` (Windows: `C:\Users\<사용자>\.ssh\deploy-key-infra-kms`)
- 용도: OCI 운영 서버(`130.162.139.142:22022`) ubuntu 사용자 접속
- 노출 시 위험: 운영 서버 root에 가까운 권한 직접 접근 (배포 가능, 데이터 조작 가능)

## ❌ 절대 사용하지 말 것

| 채널 | 이유 |
|---|---|
| 이메일 첨부 | 평문 전송, 메일 서버에 영구 보관, 차단 가능 |
| 일반 슬랙/카카오톡 메시지 | 채널/대화 노출 위험, 영구 로그 |
| 구글 드라이브 / OneDrive 일반 공유 | 링크 유출 시 누구나 접근, 검색 인덱스 노출 위험 |
| GitHub commit/PR | 영원히 git history에 박힘 (revert해도 추적 가능) |
| USB 메모리를 책상에 둠 | 분실/도난 위험 |
| 사진 찍어 메신저 전송 | 클라우드 자동 백업으로 의도치 않은 노출 |

## ✅ 권장 채널 (택1)

### 🥇 옵션 A: 1Password / Bitwarden Secure Note (가장 권장)
1. 전달자가 본인 1Password 보관함에 두 파일을 Secure Note로 저장
2. 인계받는 분의 1Password 계정에 **공유** (One-time 또는 영구)
3. 인계 완료 후 공유 해제 또는 비밀번호 회전

장점:
- 전 과정 암호화
- 만료 시간 설정 가능
- 받는 분이 바로 사용 가능
- 감사 로그 남음

### 🥈 옵션 B: SSH로 인계자 PC에 직접 복사
인계자가 본인 PC에서 운영 서버에 (전임자의 SSH 키로 한 번만) 접속해 직접 복사:

```bash
# 인계자 PC에서 (전임자가 SSH 키 한 번 빌려준 직후)
scp -i ~/.ssh/deploy-key-infra-kms -P 22022 \
  ubuntu@130.162.139.142:/opt/neolab-kms/.env \
  ./.env

# .env 받은 후 위치 이동
mv ./.env <프로젝트경로>/.env
chmod 600 <프로젝트경로>/.env
```

SSH 키 자체는:
- 전임자가 인계자 본인의 USB/직접 만나서 전달
- 또는 1Password 등으로

### 🥉 옵션 C: 임시 암호화 파일 + 별도 채널 비밀번호
1. 전임자가 두 파일을 zip + AES-256 암호화:
   ```bash
   # OpenSSL로 암호화 (Windows/Mac/Linux)
   tar czf secrets.tar.gz .env deploy-key-infra-kms
   openssl enc -aes-256-cbc -salt -pbkdf2 -in secrets.tar.gz -out secrets.tar.gz.enc
   # → 비밀번호 입력 (강력하게)
   rm secrets.tar.gz   # 평문 즉시 삭제
   ```
2. 암호화된 `secrets.tar.gz.enc` 파일은 **이메일/슬랙도 OK** (이미 암호화됐으니)
3. 비밀번호는 **다른 채널**(전화·SMS·구두) 로 전달
4. 인계자가 복호화:
   ```bash
   openssl enc -d -aes-256-cbc -pbkdf2 -in secrets.tar.gz.enc -out secrets.tar.gz
   tar xzf secrets.tar.gz
   ```

## 🎯 인계 직후 반드시 할 일 (보안 강화)

전달 끝났다고 끝이 아닙니다. 다음을 신속히 진행:

### 1. SSH 키 회전 (권장, 30분 작업)
운영 서버에 인계자의 새 SSH 키를 추가하고, 전임자의 키는 제거:

```bash
# 인계자 본인 PC에서 새 키 생성
ssh-keygen -t ed25519 -f ~/.ssh/neolab-kms-new -C "후임자@neolab.net"

# 새 공개키 출력
cat ~/.ssh/neolab-kms-new.pub
```

전임자(또는 인계자가 옛 키로 마지막 한 번) 운영 서버 접속 후:

```bash
ssh -i ~/.ssh/deploy-key-infra-kms -p 22022 ubuntu@130.162.139.142

# authorized_keys 백업
cp ~/.ssh/authorized_keys ~/.ssh/authorized_keys.bak.$(date +%Y%m%d)

# 새 키 추가
echo "<인계자의 공개키 전체 한 줄>" >> ~/.ssh/authorized_keys

# (선택, 신중) 옛 키 제거 — 인계자가 새 키로 접속 검증 후
# nano로 ~/.ssh/authorized_keys 열어서 옛 라인 삭제
```

### 2. SESSION_SECRET 회전 (선택)
.env 의 SESSION_SECRET이 노출됐을 가능성을 차단:
1. `.env`에서 SESSION_SECRET을 새 랜덤 문자열로 변경
2. PM2 재시작 → 모든 사용자 강제 재로그인
3. 다소 불편하지만 인계 직후 1회 수행 권장

```bash
# 새 시크릿 생성
node -e "console.log(require('crypto').randomBytes(48).toString('base64'))"
```

### 3. Google OAuth 클라이언트 시크릿 회전 (선택, 더 강한 보안)
Google Cloud Console에서:
1. 해당 OAuth 2.0 클라이언트 → "Reset secret"
2. 새 secret을 `.env`에 반영 → PM2 재시작

## ✅ 인계 후 정리 체크리스트

- [ ] 인계자가 새 SSH 키로 운영 서버 접속 성공
- [ ] 인계자가 본인 PC에서 `.env` 배치 후 `npm start` 정상 동작 확인
- [ ] 인계자가 변경 작업 → push → 운영 서버 pull → pm2 restart 흐름 1회 성공
- [ ] (권장) 운영 서버 `authorized_keys`에서 전임자 옛 키 라인 제거
- [ ] (권장) SESSION_SECRET 회전 후 PM2 재시작
- [ ] (선택) Google OAuth client secret 회전
- [ ] 전임자 PC에서 `.env`, `~/.ssh/deploy-key-infra-kms`를 안전 삭제(+휴지통 비우기)
  ```bash
  # Windows PowerShell
  Remove-Item -Force <프로젝트경로>\.env
  Remove-Item -Force $env:USERPROFILE\.ssh\deploy-key-infra-kms

  # Mac/Linux
  shred -u ~/<프로젝트경로>/.env  # 또는 rm -f
  shred -u ~/.ssh/deploy-key-infra-kms
  ```

체크리스트 모두 통과하면, 보안적으로도 깔끔하게 인계 완료입니다.

## 🚨 만약 노출되었다면

`.env` 또는 SSH 키가 의도치 않게 어딘가 올라가거나 유출 의심 시:

1. **즉시 운영 서버의 `authorized_keys`에서 해당 키 라인 삭제**
2. **SESSION_SECRET 회전** + PM2 재시작 → 모든 세션 무효화
3. **Google OAuth client secret 회전** + `.env` 갱신
4. **PM2 로그 검토**: `tail -200 /opt/neolab-kms/logs/$(date +%Y-%m-%d).log` 에서 `[ACCESS]` 라인의 비정상 IP/사용자 확인
5. **데이터 백업 검토**: `/opt/neolab-kms/backups/` 의 자동백업과 비교해 변경 사항 추적
