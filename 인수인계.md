# NeoLAB KMS 인수인계 매뉴얼

이 문서를 처음 받으신 분께 — **순서대로 따라하시면 1시간 이내에 동일한 작업 환경을 구축**하실 수 있습니다.

## 📋 프로젝트 개요

- **이름**: NeoLAB Guidebook (사내 지식포탈, KMS)
- **운영 URL**: https://kms.neolab.net
- **기술 스택**: Node.js 20 + Express + SQLite (better-sqlite3) + PM2 + nginx + Google OAuth
- **저장소**: https://github.com/neolab-convergence/neolab-kms
- **운영 서버**: OCI 인스턴스 (Ubuntu 22.04 ARM)

## 🚀 2단계: 저장소 클론 + 의존성 설치

```bash
git clone https://github.com/neolab-convergence/neolab-kms.git
cd neolab-kms
npm install
```

## 🔐 3단계: 민감 파일 배치

전임자로부터 안전한 채널로 전달받은 파일 두 개:

### `.env` 파일
```bash
# 받은 .env 파일을 프로젝트 루트에 배치
mv ~/Downloads/.env ./.env
```

내용 예시 (실제 값은 전달받은 것 사용):
```env
PORT=3000
SESSION_SECRET=<랜덤 문자열>
GOOGLE_CLIENT_ID=...
GOOGLE_CLIENT_SECRET=...
GOOGLE_CALLBACK_URL=https://kms.neolab.net/auth/google/callback
ALLOWED_DOMAIN=neolab.net
ADMIN_EMAILS=...
```

### SSH 키 배치
```bash
# Windows (PowerShell)
copy deploy-key-infra-kms $env:USERPROFILE\.ssh\deploy-key-infra-kms

# Mac/Linux
cp deploy-key-infra-kms ~/.ssh/deploy-key-infra-kms
chmod 600 ~/.ssh/deploy-key-infra-kms
```

### SSH 접속 테스트
```bash
ssh -i ~/.ssh/deploy-key-infra-kms -p 22022 ubuntu@130.162.139.142
# 접속 성공하면 ubuntu@... 프롬프트가 나옴. exit로 빠져나오기.
```

## 🖥️ 4단계: 로컬 실행 (선택)

서버 코드를 본인 PC에서 실행해보려면:
```bash
npm start
# 또는
node server.js
```

http://localhost:3000 에서 접속 가능. 단, OAuth 설정이 production용이라 로컬 로그인은 제한될 수 있음.

대부분의 작업은 코드 수정 → main push → 운영 서버 배포로 충분합니다.

## 🚢 5단계: 배포 흐름

### 표준 배포 절차 (5분 이내)

```bash
# 1) 로컬에서 코드 수정 후
git add <변경 파일들>
git commit -m "<메시지>"
git push origin main           # main 브랜치에 직접 push

# 2) 운영 서버 SSH 접속

# 3) 운영 서버에서 갱신
cd /opt/neolab-kms
git pull origin main
pm2 restart kms

# 4) 검증
pm2 status                      # online 상태 확인
pm2 logs kms --lines 30        # 에러 없는지 확인
```

또는 한 줄로:
```bash
ssh -i ~/.ssh/deploy-key-infra-kms -p 22022 ubuntu@130.162.139.142 \
  'cd /opt/neolab-kms && git pull origin main && pm2 restart kms'
```

### ⚠️ 캐시버스터 관리 (중요)

`public/index.html`의 CSS/JS link는 모두 `?v=NNN` 캐시버스터를 갖고 있습니다.
**프론트엔드(public/) 변경 시 반드시 +1**:

```html
<link rel="stylesheet" href="css/style.css?v=126">
<script src="js/core.js?v=126"></script>
...
```

빠른 일괄 변경:
```bash
# 예: 126 → 127로 일괄 변경
sed -i 's/?v=126/?v=127/g' public/index.html
```

## 📚 주요 기능 사용 가이드

### 개선요청 처리 상태 관리
- 관리자 모드 → 💡 개선요청
- 필터 탭 (전체/진행/완료) + 상태 배지
- "✅ 완료 처리" 버튼으로 처리자/처리일 자동 기록 (히스토리 보존)
- "↩ 다시 열기"로 진행 상태로 복귀
- "📝 메모"로 처리 메모 자유 편집
- "✕ 삭제"는 영구 삭제 (강화된 경고)

### 연락처 Excel 일괄 업로드
- 관리자 모드 → 인사 및 채용 → 연락처 → 상단 패널
- "📋 샘플 다운로드" 또는 "📤 현재 연락처 Excel 다운로드"
- Excel에서 편집 후 "📥 업로드 및 적용"
- 컬럼: `이름·직급·부서·내선번호·휴대폰·이메일·상태`
- **업로드는 항상 전체 교체** — 부분 수정은 모달 사용 권장

### 이미지 게시물 — 클릭 링크
- 관리자 모드 → 게시물 등록/수정 → 입력 방식: "이미지 (복수 업로드)"
- 각 이미지 카드 아래 URL 입력칸이 있음 (선택)
- URL 채워진 이미지는 클릭 시 새 탭으로 그 URL 이동, 빈 URL은 라이트박스 확대
- 수정 모드 진입 시 기존 이미지가 "기존" 배지와 함께 자동으로 미리보기에 채워짐
- 새 이미지 추가는 APPEND, ✕ 버튼으로 개별 제거, ◀▶로 순서 변경

## 🐛 트러블슈팅

### "변경 사항이 라이브에 반영 안 돼요"
1. 캐시버스터 `?v=N`을 +1 했는지 확인
2. 브라우저에서 Ctrl+Shift+R (강력 새로고침)
3. iOS Safari는 캐시가 강해 query string 우회 필요: `https://kms.neolab.net/?cb=1`
4. 그래도 안 되면 PM2 재시작 확인

### "메뉴 순서를 바꿔도 반영이 안 돼요"
- 라우팅 충돌 패턴 — `routes/boards.js`나 `routes/categories.js`에서 reorder 라우트가 `:id` 라우트보다 위에 등록되어 있어야 함
- Express는 첫 매칭 우선이라 `/api/boards/reorder`를 `:id="reorder"`로 잘못 잡으면 404

### "접속 기록 날짜가 하루 이전으로 보여요"
- `new Date().toISOString().split('T')[0]`은 UTC라 KST 자정~오전 9시 사이 접속이 전날로 기록되는 버그
- 모든 날짜는 `new Date().toLocaleDateString('en-CA', { timeZone: 'Asia/Seoul' })` 사용
- 새 코드 작성 시 절대 toISOString 패턴 쓰지 말 것

### "갤러리/리스트가 게시물 상세 뒤에 그대로 보여요" (모바일)
- 모바일 CSS의 `display: flex !important`가 inline `style.display='none'`을 덮어씀
- 해결: `setProperty('display', 'none', 'important')` 사용
- viewer.js의 openProductDetail 참고

### "검색 결과 클릭이 안 먹혀요"
- `goToBoardAndOpen`이 `menuItem.click()` 사용하면 expanded 메뉴는 토글로 collapse됨
- 해결: `navigateTo(boardId)` 직접 호출 (이미 적용됨)

### "PM2가 자꾸 재시작돼요" (메모리 OOM)
- `ecosystem.config.js`의 `max_memory_restart: '2048M'` 초과 시 재시작
- 메모리 누수 의심 시 `pm2 logs kms` + `pm2 monit`

### 자주 쓰는 디버깅 패턴
```bash
# 라이브 사이트 응답 헤더 (캐시 적용 여부)
curl -sI https://kms.neolab.net/login.html | grep -iE "cache|etag"

# 라이브 SQLite 컬럼 확인

```

## 🔄 데이터 복구 시나리오

### 게시물/연락처/개선요청을 실수로 삭제한 경우
1. 자동 백업 파일에서 복원: `/opt/neolab-kms/backups/auto_backup_YYYY-MM-DD.json`
2. SQLite WAL 파일에서 흔적 추적: `/opt/neolab-kms/data/kms.db-wal`
3. 매일 자정에 자동 백업되므로 자정 이후 작성→삭제는 복구 불가

복구 스크립트 예시 (`/tmp/restore.js`):
```js
const fs = require('fs');
const Database = require('better-sqlite3');
const bk = JSON.parse(fs.readFileSync('/opt/neolab-kms/backups/auto_backup_2026-05-06.json', 'utf8'));
const db = new Database('/opt/neolab-kms/data/kms.db');
const stmt = db.prepare('INSERT INTO suggestions (...) VALUES (...)');
// ... 복구 로직
```

자세한 사례는 git history의 5/7 커밋들 참고.
